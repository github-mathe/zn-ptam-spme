# zn-ptam-spme

MATLAB implementation of a Single Particle Model with Electrolyte (SPMe) for Zn--PTAm dual-ion polymer batteries.

This repository is intended to support the code release for the associated research paper. The goal is to make the implementation easy to install, run, and cite.

## Recommended Repository Structure

This workspace does not contain the MATLAB source yet, so use the structure below when you import it. Keep code, inputs, and generated outputs separate:

```text

		processed/

		figures/
		tables/

```

If the codebase already exists elsewhere, move the MATLAB functions into `code/` or a MATLAB package folder and keep the top level clean. A practical mapping for a research codebase is:

- `code/model/`: SPMe equations, parameter handling, solver setup, and state updates.
- `code/utils/`: reusable helpers for interpolation, scaling, loading, and validation.
- `code/plotting/`: scripts that regenerate the paper figures and tables.
- `examples/`: one short script per published case or experiment.
- `data/raw/`: original experimental or simulation inputs.
- `data/processed/`: cleaned inputs created from the raw data.
- `results/`: outputs generated by running the examples; do not commit large generated files.
- `docs/`: notes, derivations, or supplementary explanation that are helpful but not required for execution.

## MATLAB Source Layout

Keep the MATLAB code organized by responsibility rather than by paper section:

- `code/main.m`: the single entry point that launches the full workflow.
- `code/model/`: governing equations, solvers, parameter definitions, and simulation state updates.
- `code/utils/`: reusable helpers such as data loading, interpolation, validation, and formatting.
- `code/plotting/`: figure generation and any code that reproduces the paper visuals.
- `examples/`: short scripts for the paper cases or quick smoke tests.

If you use MATLAB package folders, a package such as `code/+spme/` is also fine. In that case, keep shared functions inside the package and expose only a small number of public entry points.

The goal is to make `code/main.m` or a single example script run the published workflow from a clean MATLAB session.

## Migration Checklist

If you are copying an existing MATLAB project into this repository, proceed in this order:

1. Identify the main script or function that reproduces the paper.
2. Move all custom functions used by that entry point into `code/`.
3. Put paper reproduction scripts in `examples/`.
4. Move input data into `data/raw/` or `data/processed/`.
5. Save regenerated figures and tables under `results/`.
6. Remove hard-coded absolute paths and local machine settings.
7. Run the main example from a fresh MATLAB session to confirm it works.
